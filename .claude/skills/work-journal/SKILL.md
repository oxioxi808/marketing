# Work Journal — оперативная память проекта между сессиями

Скилл ведёт **`journal/work-journal.md`** в репо — единый файл, с которого начинается каждая сессия и которым она заканчивается. Цель: не переспрашивать пройденное и не терять решения между сессиями и между машинами.

## 1. Где живёт журнал
- Файл: **`journal/work-journal.md`** в корне репо. Если его нет — **создать** (вместе с папкой `journal/`).
- Язык — как у проекта (русский; термины на английском).

## 2. Структура файла (строго сверху вниз)

```
# WORK JOURNAL — <проект>

## ТОЧКА ВОЗВРАТА
Текущий этап: <N — название>.
Следующие 3–5 конкретных действий (с исполнителем):
1. [Оксана] …
2. [Code] …
3. [AI Studio] …
4. [Claude.ai-чат] …

## РЕШЕНИЯ
<новые записи — СВЕРХУ>
- <YYYY-MM-DD> — <суть решения> — в каноне: да ✅ (<файл>, <коммит>) | нет
- …

## СТАТУС НА КОНЕЦ СЕССИИ
Этапы: <какие закрыты / текущий>.
Последний коммит: <хеш + краткое имя>.
Блокеры: <список или «нет»>.
```

**Исполнители в ТОЧКЕ ВОЗВРАТА** помечать явно: `[Оксана]` (руками), `[Code]` (Claude Code — этот интерфейс), `[AI Studio]` (генерация контента), `[Claude.ai-чат]` (обсуждение/решения).

## 3. Триггеры обновления (автоматически, без запроса пользователя)
1. **Коммит, меняющий канон** (`offers/*`, `pricing*`, `listings*`, `plans/*`) → добавить запись в **РЕШЕНИЯ тем же коммитом** (со ссылкой на файл и хеш).
2. **Конец сессии** или команды **«закрываем» / «обнови журнал»** → обновить **ТОЧКУ ВОЗВРАТА** и **СТАТУС**, затем **commit + push** (по git-правилу проекта: прямо в main).
3. **Решение из Claude.ai-чата**, принесённое сюда → внести в РЕШЕНИЯ со статусом **«в каноне: нет»**, затем провести в канон-файлы и **обновить статус на «да ✅ (файл, коммит)»**.

## 4. Правило записи
- **Новое — сверху** (в РЕШЕНИЯХ обратный хронологический порядок).
- Осевшее в каноне — **пометить** «в каноне: да ✅ (файл, коммит)».
- Устаревшее/отменённое — **вычеркнуть** `~~строка~~`, **не удалять** (сохраняем историю).

## 5. Начало новой сессии
- **Первое действие — прочитать `journal/work-journal.md`.**
- Продолжать **с ТОЧКИ ВОЗВРАТА**, не переспрашивая уже пройденное.

## 6. Журнал — НЕ канон
- При расхождении журнала с канон-файлами (`offers/*`, `pricing*`, `plans/*`) **правы канон-файлы**.
- Расхождение чинится **записью в журнал** (журнал подтягивается к канону), а не правкой канона под журнал.

## Порядок действий при срабатывании триггера
1. Прочитать текущий `journal/work-journal.md` (или создать по шаблону §2, если нет).
2. Внести изменение по нужному триггеру (§3) и правилу записи (§4).
3. Если триггер требует push (§3.2) — `git add journal/work-journal.md` → commit → push в main.
4. Не превращать журнал в канон: детали офферов/цен/плана живут в своих файлах; в журнале — указатели (файл + коммит) и точка возврата.
